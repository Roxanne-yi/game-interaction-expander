# Contract-First Workflow

Use this reference before producing any Figma board. The workflow exists to keep analysis, 3.0 flows, 4.0 wireframes, notes, and Figma rendering aligned without letting renderer code replace LLM's design work.

## Rule Ownership

- Execution rules live in Markdown references.
- Result rules live in `contracts/*.contract.json`.
- Renderer geometry and template slots live in `assets/krad-template/template-manifest.json`.
- Scripts enforce ordering and deterministic validation.
- LLM owns semantic conversion and design expression; scripts own scaffold, style locks, transport, and measurable validation.

Do not duplicate detailed result rules in prompts. Load the relevant contract and use the validator.

## Artifacts

Create a run directory per generation attempt:

```text
run-dir/
  run-state.json
  source-analysis.json
  source-analysis.validation.json
  interaction-model.json
  interaction-model.validation.json
  analysis-coverage.validation.json
  render-payload.json
  figma-scripts/
    figma-render.payload-000-bootstrap.generated.js
    figma-render.payload-001-overview.generated.js
    figma-render.payload-002-flows.generated.js
    figma-render.payload-003-feature-module-001-start.generated.js
    figma-render.payload-004-feature-module-001-row-001.generated.js
    ...
    figma-render.payload-999-finalize.generated.js
    figma-validator.generated.js
    figma-gate.generated.js
    figma-scripts.index.json
  figma-execution-state.json
```

`source-analysis.json` is the structured PRD reading pass. `interaction-model.json` is the semantic source of truth before Figma rendering. `analysis-coverage.validation.json` proves the model did not drop source facts before render. `render-payload.json` is the only allowed renderer input and must be generated by `scripts/build-render-payload.js`.

## Stage Order

1. `initialized`
   - Created by `node scripts/run.js init <run-dir> <source-analysis.json> <interaction-model.json>`.
   - Copies source analysis and the model into the run directory.

2. `source_analysis_validated`
   - Created by `node scripts/run.js validate-source <run-dir>`.
   - Runs `scripts/validate-source-analysis.js`.
   - Blocks on thin source sections, missing evidence, damaged Chinese text, and malformed source ids.

3. `model_validated`
   - Created by `node scripts/run.js validate-model <run-dir>`.
   - Runs `scripts/validate-interaction-model.js`.
   - Blocks on schema, UTF-8, id reference, flow, wireframe, note, and AI pending failures.

4. `coverage_validated`
   - Created by `node scripts/run.js validate-coverage <run-dir>`.
   - Runs `scripts/validate-analysis-coverage.js`.
   - Blocks when any source-analysis item lacks a model mapping, when mappings point to missing ids, or when interface/flow/AI pending counts are thinner than the source analysis.

5. `payload_ready`
   - Created by `node scripts/run.js build-payload <run-dir>`.
   - Runs `scripts/build-render-payload.js`.
   - Projects the validated model into the exact renderer payload shape.
   - Blocks hand-assembled or fallback payloads.

6. `render_ready`
   - Created by `node scripts/run.js render-ready <run-dir>`.
   - Requires `render-payload.json`.
   - Allows configured Figma script generation.

7. `official_render_required`
   - Created by `node scripts/run.js generate-figma-scripts <run-dir>` or `node scripts/run.js prepare <run-dir> <source-analysis.json> <interaction-model.json>`.
   - Writes direct official scaffold scripts, validator, and gate scripts under `run-dir/figma-scripts/`.
   - Every generated Figma script must be 25k characters or smaller.
  - Render payload scripts write template sections in order; no Figma-side source chunk assembly is allowed.
  - 3.0 and 4.0 scripts create editable LLM content slots from the template. They do not draw the final player-flow diagram or low-fidelity interface content with fixed generic layouts.
  - 4.0 is split by main feature module and interface row when scripts are generated.
  - Script-size limits must be solved by smaller batches, not by deleting notes, wireframe controls, states, feedback, modal, toast, or AI待确认 content.
  - These generated scripts are the only approved Figma execution inputs for the run.
   - `node scripts/run.js figma-next <run-dir>` returns `codeFile`, `chars`, `bytes`, and `transferRule`; it does not return inline `code` by default.
   - Read `codeFile` through nodeRepl/read-payload and pass the file contents, not the path, to `use_figma.code`.
   - Large payload size is expected for direct transfer; never skip, downgrade, rewrite, split again, or fallback because of `chars` / `bytes`.
   - Use `node scripts/run.js figma-next <run-dir> --inline-code` only when direct file-to-`use_figma.code` transfer is unavailable.
   - `node scripts/run.js figma-record <run-dir> --status pass --board-id <boardId>` records success and advances `figma-execution-state.json`.
   - If payload transport or execution fails, report `renderer_transport_failed` and stop. Never create a fallback board.

Use `node scripts/run.js status <run-dir>` to check the next legal step.

## Model Shape

The model uses generic ids and contracts, never project-specific hardcoded names:

- Source-analysis items use `SA-MECH-001`, `SA-IF-001`, `SA-FLOW-001`, `SA-STATE-001`, `SA-RISK-001`, and `SA-AI-001`.
- Major modules use `MOD-001`.
- Interfaces use `IF-001`.
- Flow nodes use `NODE-001`.
- Wireframe elements use `EL-*`.
- Flow edges reference node ids.
- Notes reference `targetElementId`.
- `sourceCoverage[]` maps every source-analysis item to one or more concrete model ids.

Concrete feature content belongs in model values, not in contracts.

## 3.0 Flow Contract

Use `contracts/flow.contract.json`.

Each flow has:

- `flowId`
- `title`
- `nodes[]`
- `edges[]`

Every edge has:

- `from`
- `to`
- `label`
- `relationshipType`
- `evidence`

Allowed relationship types are `navigation`, `stateChange`, `systemFeedback`, `branchDecision`, `returnPath`, and `externalSurface`.

The renderer may lay out nodes, but it must not infer player logic from template connector order.

## 4.0 Wireframe And Notes Contract

Use `contracts/wireframe.contract.json`, `contracts/note.contract.json`, and `contracts/ai-pending.contract.json`.

Every interface contains:

- A stable `id`.
- A structured design brief for the low-fidelity wireframe.
- `primaryTask` and player-visible regions/elements so LLM can draw the interface and notes can target the design.
- Element ids on regions, controls, states, and feedback items.
- Notes that target those element ids.

The official scaffold must create every interface row as content-driven layout:

- Left wireframe is fixed to the template player-screen frame.
- Right-side notes can grow vertically according to content.
- Row height is the larger of the wireframe and notes, plus spacing.
- Later rows, later modules, footer, and board height must move down with content growth.

LLM then draws inside the scaffold:

- 3.0 player-flow diagrams by real player goals and edges.
- 4.0 player-visible wireframes inside `llm wireframe slot <interfaceId>`.
- 4.0 designer-facing notes inside `llm note slot <interfaceId>` using template labels and numbered text.
- 3.0 node thumbnails should reuse or visually match the 4.0 wireframe for the referenced interface/state.

Right-side notes must include:

- `tag`
- `targetElementId`
- `body`
- `basis`
- `evidence`
- `coverage[]`, covering purpose, module composition, key states, player actions, post-action changes, feedback, and AI pending.

Every major module needs at least one visible `AI待确认` note. AI pending notes target concrete elements or states and use `inferredGap`, `sourceConflict`, `designRisk`, or `ruleGenerated` as basis.

## Encoding

All run artifacts must be UTF-8 without BOM and LF. The validator rejects:

- Replacement characters.
- Question-mark replacement blocks.
- Likely mojibake from damaged Chinese text.

When using Windows shell, prefer PowerShell and explicit `-Encoding utf8`. Do not pass raw Chinese JSON through command strings.
